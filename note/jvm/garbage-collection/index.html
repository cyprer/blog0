    垃圾回收机制¶

    引用类型¶

    引用类型有四种:

    1. 强引用,被强引用关联的对象不会被回收。
    2. 软引用,被软引用关联的对象只有在内存不足时才会被回收。
    3. 弱引用,被弱引用关联的对象无论内存是否充足都会被回收。
    4. 虚引用,虚引用关联的对象和没有引用关联的对象一样,任何时候都可能被回收。

    垃圾回收算法¶

    1. 标记-清除算法

    标记:从 GC Roots 对象开始，遍历整个对象图，标记所有存活的对象。(注意，标记的是存活对象，不是垃圾对象)

    清除:遍历整个堆内存，回收所有未被标记的（即死亡的）对象所占用的空间。

    优点是速度快,缺点是会产生内存碎片

    1. 复制算法 复制:将所有对象复制到另一块内存中，然后将原内存空间回收。 优点是不产生内存碎片,缺点是浪费了一半的内存空间,一定的性能差

    2. 标记-整理算法 标记：它的第一个阶段与标记-清除算法是一模一样的，均是遍历 GC Roots，然后将存活的对象标记。

    整理：移动所有存活的对象，且按照内存地址次序依次排列，然后将末端内存地址以后的内存全部回收。因此，第二阶段才称为整理阶段。

    优点是不会产生内存碎片,缺点就是速度最慢

    对象存活的判断方法¶

    1. 引用技术法 引用技术法就是通过判断对象是否被其他对象所引用来判断对象是否存活。 缺点就是无法解决循环引用问题
    2. 可达性分析 这个算法的基本思想就是通过一系列的称为 “GC Roots” 的对象作为起点，从这些节点开始向下搜索，节点所走过的路径称为引用链，当一个对象到 GC Roots 没有任何引用链相连的话，则证明此对象是不可达的。 但是，并不是说当进行完可达性分析算法后，即可证明某对象可以被GC。对象是否存活，需要两次标记：第一次标记通过可达性分析算法。如果没有GC Roots相连接的引用链，那么将第一次标记如果对象的finalize()方法被重写并且没有执行过，则放在F-Queue队列中等待执行(不一定会执行)，如果一段时间后该对象的finalize()方法被执行且和GC Roots关联，自救成功。如果仍然没有关联，则进行第二次标记，才会对该对象进行回收 可达性分析的一大缺点就是执行时间过长,整个过程是STW的,因此为了解决这个问题,三色标记法算法应运而生

      三色标记法算法(判断垃圾的算法) 白色：该对象没有被标记过。在垃圾回收周期开始时，所有对象都是白色。这代表着它们是潜在的垃圾

    灰色：该对象已经被标记过了，但该对象的引用的对象还没标记完。

    黑色：该对象已经被标记过了，并且他的全部引用对象也都标记完了。 过程:
    三色标记法的标记过程可以分为三个阶段：初始标记（Initial Marking）、并发标记（Concurrent Marking）和重新标记（Remark）。

    初始标记： 所有对象初始都是白色。遍历所有的根对象，将根对象和直接引用的对象标记为灰色。在这个阶段中，垃圾回收器只会扫描被直接或者间接引用的对象，而不会扫描整个堆。因此，初始标记阶段的时间比较短。（Stop The World）

    并发标记：在这个过程中，垃圾回收器会从灰色对象开始遍历整个对象图，将被引用的对象标记为灰色，并将已经遍历过的对象标记为黑色。并发标记过程中，应用程序线程可能会修改对象图，因此垃圾回收器需要使用写屏障（Write Barrier）技术来保证并发标记的正确性。（不需要STW）

    重新标记：重新标记的主要作用是标记在并发标记阶段中被修改的对象以及未被遍历到的对象。这个过程中，垃圾回收器会从灰色对象重新开始遍历对象图，将被引用的对象标记为灰色，并将已经遍历过的对象标记为黑色。（Stop The World）

    并发在重新标记阶段结束之后，垃圾回收器会执行清除操作，将未被标记为可达对象的对象进行回收，从而释放内存空间。 - 三色标记法的问题 1. 并发标记过程中的浮动垃圾问题
    2. 漏标问题:并发标记中一个黑色对象突然引用了一个白色对象

    垃圾收集器¶

    按执行方式分类

    1. 串行垃圾收集器:Serial+Serial Old

    2. 并行垃圾收集器:Parallel Scavenge+Parallel Old+ParallelNew

    3. 并发垃圾收集器:CMS、G1、ZGC 按回收区域分类

    4. 新生代垃圾回收器:Parallel Scavenge+ParallelNew

    5. 老年代垃圾回收器:Serial Old+Parallel Old+CMS

    6. 整堆回收器:G1+ZGC

    串行垃圾收集器¶

    • Serial GC 采用的复制算法,收集新生代垃圾,不存在内存碎片,会STW,性能差
    • Serial Old GC 采用的标记-整理算法,收集老年代垃圾,不会存在内存碎片,也会STW,性能差

    并行垃圾收集器¶

    • Parallel New GC : 其实就是Serial GC的多线程版本使用的也是复制算法

    • Parallel Scavenge GC：采用复制算法，仅收集新生代垃圾，不存在内存碎片，多线程并行执行GC，全程STW（STW时长比Serial GC短），性能优于Serial GC

    • Parallel Old GC：采用标记-整理算法，仅收集老年代垃圾，不存在内存碎片，多线程并行执行GC，全程STW（STW时长比Serial Old GC短），性能优于Serial Old GC

    并发垃圾收集器¶

    • CMS (Concurrent Mark Sweep) GC：采用标记-清除算法，CMS 只负责老年代，新生代仍用 ParNew（复制算法），两者配合工作，会产生内存碎片，分初始标记、并发标记、重新标记、并发清除四个阶段（仅初始标记和重新标记阶段STW，并发标记/清除阶段与用户线程并行），STW总时长较短，但依然存在一些问题,就是会有浮动垃圾和内存碎片
    • G1 (Garbage-First) GC：兼顾新生代和老年代收集，整体采用标记-整理算法，局部（新生代）采用复制算法，堆内存划分：抛弃 “新生代 / 老年代” 的物理隔离，将整个堆划分为多个大小相等的 Region,无内存碎片，分初始标记、并发标记、最终标记、筛选回收四个阶段。最终标记:处理并发标记的漏标问题;筛选回收:优先选择垃圾最多的 Region，将存活对象复制到空 Region，原 Region 清空（复制算法）；仅初始标记和筛选回收阶段会STW
    • ZGC (Z Garbage Collector)：面向大内存场景，采用着色指针+读屏障技术，整体基于标记-整理算法，无内存碎片，收集过程中仅在初始标记和最终标记阶段有极短STW（微秒级），其余阶段完全并发，支持TB级内存，吞吐量和延迟表现优异（JDK 11引入，JDK 15正式发布）。 我帮你把这个核心区别清单完善成面试级、逻辑清晰、用词精准的版本，重点突出“主要区别”，不冗余：

    CMS和G1的主要区别¶

    1. 收集区域与内存布局¶

    • CMS：仅负责老年代的垃圾收集，新生代需搭配ParNew收集器（复制算法）协同工作；堆内存采用「物理隔离」设计，分为连续的新生代（Eden+S0+S1）和连续的老年代两大块，代别固定、内存地址连续。
    • G1：覆盖全堆（新生代+老年代） 收集，无需搭配其他收集器；堆内存抛弃“新生代/老年代物理隔离”的设计，整体划分为多个大小相等（512KB~32MB可配置）的独立Region，Region可动态标记为Eden、Survivor、Old或Humongous（专门存储大对象），代别仅为逻辑标记，可随GC动态调整。

    2. 并发标记修正机制（核心解决“并发标记时引用变动导致漏标”）¶

    CMS：增量更新（Incremental Update）¶

    • 核心背景：CMS是老年代并发收集器，并发标记阶段（JVM一边标记存活对象、一边运行业务代码），业务线程可能删除老年代对象的引用关系，导致存活对象被“漏标”，最终错误回收。
    • 实现逻辑：
    • 并发标记阶段，通过「写屏障」监控所有老年代对象的引用删除操作（如A.ref = null，A取消引用B）；
    • 不直接处理被引用对象B，而是将“引用源对象A”标记为「脏对象」，加入脏对象队列；
    • 重新标记阶段（STW），遍历脏对象队列中的所有对象（如A），重新扫描其完整引用链，补充标记漏标的存活对象（如B）；
    • 仅扫描“引用变动的脏对象”，而非全量老年代，保证重新标记的STW时间可控。
    • 优缺点： ✅ 优点：标记精准，几乎不会“错留垃圾”，内存利用率高； ❌ 缺点：重新标记阶段需扫描脏对象的完整引用链，若脏对象数量多，STW开销会显著增加；适配小堆（4-8GB）场景，大堆下STW时间易失控。

    G1：SATB（Snapshot At The Beginning，初始快照）¶

    • 核心背景：G1是全堆分区化收集器，并发标记需覆盖所有Region，若沿用增量更新，大堆下脏对象引用链扫描会导致STW超时，因此设计SATB适配“可控停顿”目标。
    • 实现逻辑：
    • 并发标记开始时，为全堆对象的引用关系生成「快照」（记录此时所有对象的引用状态）；
    • 并发标记阶段，通过「写屏障」监控所有对象的引用删除操作（如A.ref = null）；
    • 检测到引用删除时，不标记引用源A，而是直接将「被引用对象B」加入待处理队列；
    • 最终标记阶段（STW），无需扫描引用链，直接将待处理队列中的B标记为存活（按快照状态判定）；
    • 即使B实际已无引用，本次GC也会保留，待下次GC重新标记后再回收。
    • 优缺点： ✅ 优点：无需扫描引用链，最终标记的STW开销极低，适配8GB+大堆的“可控停顿”需求； ❌ 缺点：可能“错留少量垃圾”（本次GC未回收实际无引用的B），临时占用内存，但下次GC会修正，无内存泄漏风险。

    3. 跨代/跨Region引用处理（核心解决“GC时扫描范围过大导致STW过长”）¶

    CMS：仅依赖卡表（Card Table）¶

    • 核心背景：新生代GC（Minor GC）时，需找到「老年代对象引用的新生代对象」（这些对象是存活的，不能回收），若遍历整个老年代，STW时间会极长。
    • 实现逻辑：
    • 将老年代内存划分为多个512KB的「卡页」，每个卡页对应卡表（字节数组）中的一个元素；
    • 当老年代对象引用新生代对象时，通过「写屏障」将对应卡页标记为「脏卡」；
    • 新生代GC时，仅扫描卡表中的「脏卡」，无需遍历整个老年代，快速找到跨代引用的新生代对象。
    • 适用范围：仅解决“老年代→新生代”的跨代引用，无跨区域引用的处理能力（因CMS的老年代是连续内存，无Region划分）。

    G1：卡表（Card Table）+ RSet（Remembered Set，记忆集）¶

    • 核心背景：G1的堆是Region分区化布局，不仅有跨代引用，还有「跨Region引用」（如Old Region引用Eden Region、Region A引用Region B），仅用卡表无法定位跨Region的引用。
    • 实现逻辑：
    • 卡表：复用CMS的卡表逻辑，解决“老年代Region→新生代Region”的跨代引用，避免扫描整个老年代；
    • RSet：为每个Region维护一个RSet，记录「外部Region对本Region对象的引用」（如Region B的RSet记录“Region A的XX对象引用我Region B的YY对象”）；
    • 当外部Region的对象引用本Region对象时，通过写屏障更新本Region的RSet；
    • GC回收某个Region时，仅扫描该Region的RSet，无需遍历全堆，快速找到外部引用的存活对象。
    • 核心价值：适配Region分区化布局，将GC扫描范围从“全堆”缩小到“目标Region+RSet”，是G1实现“可控停顿”的关键基础。

    HTTPS¶

    HTTP 的安全问题¶

    1. 明文传输，容易被窃听
    2. 所有数据都是明文，抓包即可看到账号、密码、内容、Cookie

    3. 无任何机密性

    4. 报文容易被篡改

    5. 中间人可修改请求或响应内容

    6. 注入广告、恶意代码、篡改页面、劫持跳转

    7. 无法识别通信方身份，容易被冒充

    8. 不知道对方是不是真实服务器

    9. 可被钓鱼网站、中间人攻击冒充

    10. 无法防止重放攻击

    11. 截获请求后重复发送，可能导致重复下单、重复扣款

    HTTPS 解决方案¶

    HTTPS = HTTP + TLS/SSL 加密层 在应用层与传输层之间加一层安全协议，一次性解决 HTTP 四大问题。

    1. 解决【窃听】：混合加密（非对称 + 对称）¶

    早期 HTTPS 使用 RSA 非对称加密 做密钥交换：

    1. Client Hello 客户端发送 TLS 版本、加密套件、客户端随机数。

    2. Server Hello 服务器选定加密套件，返回服务器随机数。

    3. 服务器发送证书 证书中携带 RSA 公钥。

    4. 客户端验证证书 验证通过后，信任服务器公钥。

    5. 客户端生成 Pre-Master Secret 用 RSA 公钥加密 后发送给服务器。

    6. 两端使用： 客户端随机数 + 服务器随机数 + Pre-Master Secret 生成相同的对称密钥。

    7. 后续全部使用对称加密传输数据，解决明文窃听问题。

    - 这里有个问题是RSA并不能做到前向加密,也就是说如果一但服务器的私钥泄露了,过去的密文也就会被破解了,因此现在的https使用ECDHE来结局这个问题,大概就是生成临时密钥对,即使泄露也只是泄露一次会话的而已,临时公钥是根据临时私钥生成的,所以不能只使用一个公钥.¶

    2. 解决【篡改】：消息摘要 + 完整性校验¶

    • 使用 哈希算法（SHA256） 生成报文摘要
    • 结合密钥生成 MAC 消息认证码
    • 传输时把密文 + 摘要一起发送
    • 接收方解密后重新计算摘要，对比是否一致

    只要被修改 1 字节，校验直接失败，连接断开。

    → 保证数据完整性

    3. 解决【冒充】：数字证书 + CA 认证¶

    • 服务器向 CA 机构 申请数字证书
    • 证书包含：公钥、域名、有效期、CA 签名
    • 客户端（浏览器/系统）内置信任 CA 列表
    • 客户端连接时验证：
    • 证书是否由可信 CA 签发
    • 域名是否匹配
    • 是否过期/吊销

    服务器信息哈希后再用私钥加密就是签名,服务器信息 + 签名就是数字证书

    客户端收到数字证书后拿到服务器信息和签名,对服务器信息哈希和对签名解密,如果一致即可证明证书未篡改

    → 保证身份真实性，防钓鱼、防冒充

    4. 解决【重放】：随机数 + 序列号¶

    • 握手时生成客户端随机数、服务器随机数
    • 通信中每个记录都有序列号
    • 重放的包序列号重复，直接丢弃

    → 防止重放攻击

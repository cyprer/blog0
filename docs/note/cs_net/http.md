# HTTP 演变史

## 1. HTTP/0.9 —— 最原始，只够“能用”
### 特点
- 只有 **GET 请求**
- 没有请求头、响应头
- 没有状态码、没有POST、没有图片
- 一次请求，建立一次 TCP 连接，请求完立刻断开

### 问题
- 功能极弱，只能返回简单 HTML
- 无任何扩展能力
- 无状态、无错误处理

---

## 2. HTTP/1.0 —— 功能补齐，但性能很差
### 新增能力
- 新增请求头/响应头
- 新增 **POST、HEAD** 方法
- 新增状态码
- 支持图片、脚本、样式等资源
- 简单缓存：`Expires`

### 但存在 4 个致命问题
1. **短连接**
   每个请求都要重新建立 TCP 连接（三次握手），延迟极高、开销巨大。

2. **完全串行**
   请求1 → 等响应1 → 请求2 → 等响应2
   一个慢，全部慢。

3. **没有 Host 头**
   一台服务器只能对应一个域名，无法做虚拟主机。

4. **缓存机制简陋**
   只有 `Expires` 绝对时间，无法精确控制。

5. 不支持分段在下载
---

## 3. HTTP/1.1 —— 大规模商用的基石
### 目标：解决 HTTP/1.0 所有痛点

### ① 解决「短连接、开销大」
- 默认开启 **长连接（Keep-Alive / Persistent Connection）**
- 一个 TCP 连接可以发送**多个请求和响应**
- 减少大量三次握手 + 慢启动开销

### ② 解决「请求串行、效率低」
- 引入 **Pipelining（管道化）**
- 可以**连续发送多个请求**，不用等待前一个响应回来
- → **解决了“请求发送”层面的队头阻塞**

### ③ 解决「一台服务器只能一个域名」
- 新增 **Host 请求头**
- 标识要访问的域名
- 支持**虚拟主机**，一台服务器部署多个网站

### ④ 解决「缓存弱、重复传输」
- 新增 `Cache-Control`（更灵活、优先级更高）
- 新增 `ETag / If-None-Match`
- 新增 `Last-Modified / If-Modified-Since`
- 缓存更精准，减少无效请求

### ⑤ 解决「大文件无法断点续传」
- 支持 **Range 分块请求**
- 响应 `206 Partial Content`
- 支持下载中断后继续、视频拖拽

---

### HTTP/1.1 仍然存在的核心问题
1. **响应队头阻塞（最致命）**
   - Pipelining 只是**请求可以并发发**
   - 但**响应必须按请求顺序返回**
   - 第一个响应卡住，后面所有响应都被阻塞

2. **请求头越来越大，大量重复**
   Cookie、User-Agent、Origin 每次都带，浪费带宽。

3. **并发能力差**
   浏览器只能对同一个域名开 6～8 个连接，再多就排队。

---

## 4. HTTP/2 —— 彻底解决 HTTP/1.1 性能瓶颈
### 目标：解决 HTTP/1.1 慢、阻塞、开销大

### ① 二进制分帧（底层基础）
- 不再是文本协议，改为**二进制格式**
- 把请求/响应切分成很小的 **帧（Frame）**
- 每个帧带：
  - Stream ID（属于哪个请求）
 - 长度、类型、标记
- 解析更快、更紧凑、更安全

### ② 多路复用（Multiplexing）—— 真正解决队头阻塞
- **同一个 TCP 连接** 上可以同时有多个 **Stream（流）**
- 每个请求 = 一个独立 Stream
- 帧可以**乱序传输、交错发送**
- 客户端根据 Stream ID 重新组装
- → **一个流阻塞，不影响其他流**
- → **彻底解决应用层的请求 & 响应队头阻塞**

### ③ HPACK 头部压缩
- 专门压缩请求头/响应头
- 维护两端的**索引表**
- 大量重复字段只传索引
- 头部体积减少 50%～90%

### ④ 服务器推送 Server Push
- 服务器可以**主动推送资源**给客户端
- 不用等浏览器解析 HTML 后再请求
- 例如：请求 HTML → 服务器直接推 CSS、JS

---

### HTTP/2 遗留问题
- 底层仍是 **TCP**
- TCP 有**队头阻塞**：
  一个丢包，整个连接的所有流都要等重传
- 这叫 **TCP 层队头阻塞**

---

## 5. HTTP/3 —— 最终、彻底解决队头阻塞
### 目标：干掉 TCP 队头阻塞

### ① 底层换成 **QUIC 协议**（基于 UDP）
- 不再用 TCP
- UDP 不保证顺序、不保证可靠性
- QUIC 自己实现可靠性、有序、加密

### ② 真正的流隔离
- 每个流**完全独立**
- 一个流丢包 → 只重传这个流
- **不影响其他流**
- → 彻底消除 **所有层面的队头阻塞**

### ③ 0-RTT 握手
- 首次连接 1-RTT
- 再次连接 **0-RTT**
- 直接发数据，不用等握手
- QUIC内部包含了TLS
### ④ 解决了连接迁移
- HTTP/3 基于 QUIC，使用 Connection ID 标识连接，而不是依赖 IP + 端口。当设备切换网络（4G ↔ WiFi）导致 IP 变化时，连接不会断开，可直接无缝继续传输，解决了 TCP 无法跨网络保持连接的问题。
